Lab 2 — Interaction

User Interaction
▶ Watch Demo Video ⌥ GitHub Repository

For this lab, I upgraded my SDG app's static layout by adding a TextField and a Button to make it interactive. I used remember { mutableStateOf("") } to hold the input value, so that whenever the user types and taps the button, the state changes and Compose automatically recomposes the UI to show the updated message. This connected directly to the lecture concept of state hoisting keeping the state variable at the top of the Composable so both the TextField and Button could read and update it. At first it was a bit confusing to understand why the UI wouldn't update without mutableStateOf, but once I saw recomposition happen live, the concept of reactive UI in Compose finally clicked for me.

Lab 5 — Room

Local Database (Room)
▶ Watch Demo Video ⌥ GitHub Repository

For this lab, I upgraded my SDG app into a full MVVM + Repository + Room architecture by creating an Entity to represent my data, a DAO interface with insert() and getAll() returning a Flow>, a Room database as a singleton, a Repository to connect the ViewModel to the DAO, and a ViewModel that exposes the data as a StateFlow to the UI. 03 · Project Submissions

Projects

Project 1

Built on Lab 4 + Working Features
Problem Statement
ReServe addresses food waste and underused household items within local communities by connecting people who have surplus food or unused belongings with neighbors who need them. It extends the multi-screen navigation pattern from Lab 4 by using Jetpack Navigation Compose to link a home/browse screen, category screens, item detail screens (food and non-food), a cart/checkout flow, a listing-creation flow, chat, and a profile screen into one cohesive app.
Key Features
The app provides at least five connected screens home/browse, category listing, item detail (food/non-food), cart, and an add-item form linked through Navigation Compose. A single shared ViewModel (ReServeViewModel) holds app state such as the items list, cart contents, and user-added listings using StateFlow/mutableStateOf. Users can add a new item via the Add Item screen, and it immediately appears in the home/category screens and item lists, demonstrating shared state across screens. Reservation and borrowing actions update stock/availability shown on other screens.
Lessons Learned
Building ReServe within the constraints of this project taught us the value of keeping app state centralized in a single ViewModel using StateFlow and mutableStateOf, rather than reaching for a database. Designing the navigation graph for five or more screens reinforced how Navigation Compose routes and arguments let separate screens read and update the same shared state, which made the "add an item and see it elsewhere" requirement straightforward once the ViewModel was structured correctly. I also learned that working through Lab 4's navigation and state-sharing patterns first made it much easier to scale up to a full multi-screen app, since the core architecture (NavHost, ViewModel, StateFlow) stayed the same and just grew in scope.

Project 2

Lab 5 + 2 Screens + API + Sensors + Cloud
Problem Statement
Food waste and unused household items are common in communities edible surplus food gets discarded while usable non-food items sit idle, even though nearby residents could use them. ReServe addresses this by letting users list surplus food (at a discount, before it expires) and lend non-food items (tools, furniture, books) to neighbors, reducing waste and encouraging community sharing.
Key Features
The app extends the Lab 5 Room database (UserListedItem, CartItem, ChatMessage, Reservation, BorrowedItem, UserProfile entities) with new functionality across two key screens ProfileScreen and FoodDetailScreen. ProfileScreen lets users view and edit their display name, bio, location, and phone number, persisted locally via Room (UserProfileEntity/UserProfileDao). It integrates the device GPS sensor (via LocationViewModel/LocationHelper using FusedLocationProviderClient) to auto-detect and fill in the user's current location with one tap. FoodDetailScreen integrates the Open Food Facts REST API (via Retrofit/OpenFoodFactsClient) to fetch live nutrition data — calories, macros, allergens, Nutri-Score, and dietary labels — for each food listing, displayed in a NutritionCard. It also uses GPS data from LocationViewModel to show the real-time distance between the user and the item's seller. For cloud sync, FirestoreRepository uploads, updates, and deletes user-listed items in a Firebase Firestore "community_listings" collection whenever a user adds, edits, or deletes a listing (ReServeRepository.addUserItem/updateUserItem/deleteUserItem), enabling listings to be shared across devices/users beyond local Room storage.
Lessons Learned
Combining Room with Firestore required careful separation of concerns — Room remains the source of truth for fast, offline-first local reads (exposed as Flow/StateFlow to Compose), while Firestore acts as a sync layer triggered alongside local writes, with graceful degradation (returning empty lists) if the network is unavailable. Integrating a REST API (Retrofit + Gson) taught the importance of nullable, defensive data models since real-world APIs often return incomplete records — the NutritionCard handles loading, success, not-found, and error states explicitly. Using GPS sensors highlighted the need for permission handling (runtime permission requests) and fallback values when location is unavailable, ensuring the UI never breaks even without a location fix. Overall, layering local persistence, sensors, REST APIs, and cloud sync within a shared ViewModel/Repository architecture reinforced the value of unidirectional data flow and StateFlow for keeping the UI reactive and consistent across all these asynchronous data sources.

04 · Reflection & Learning Journey

From Lab 1 to Project 2

My journey began in Lab 1 where I learned the very basics of Jetpack Compose — arranging UI elements using Row, Column, Box, and Text to recreate a screen layout. It felt simple but foundational. By Lab 2, I added interactivity through TextField and Button with state management. Lab 3 pushed me further into Material Design 3, where I learned to apply consistent color schemes, typography, and theming — which directly shaped the visual identity of ReServe. Lab 4 introduced Navigation and ViewModel, teaching me how to manage multi-screen flows and share state across composables without losing data on rotation. Lab 5 was a turning point — integrating Room Database taught me the full MVVM + Repository architecture. By Project 2, I had built upon every single lab: combining Room for local persistence, Firebase Firestore for cloud sync, the Open Food Facts REST API via Retrofit for live nutrition data, and GPS via FusedLocationProviderClient for real-time location awareness. What started as a static birthday card UI had evolved into a fully connected, sensor-aware, data-persistent community sharing application.

The biggest challenge I faced was understanding how all the architectural layers connect — specifically making the Room DAO, Repository, ViewModel, and StateFlow work together as a clean pipeline without causing data loss or UI inconsistencies. Debugging Room migration errors (when I added new entities like UserProfileEntity and ReservationEntity) required learning about fallbackToDestructiveMigration() and understanding database versioning. Integrating Firebase Firestore alongside Room was another challenge — I had to ensure that local inserts and cloud syncs happened correctly without blocking the main thread, which taught me the importance of Kotlin coroutines and suspend functions. On the API side, parsing the Open Food Facts JSON response and gracefully handling null fields (since not every product has complete nutrition data) required careful state management using a sealed class. The GPS integration also had edge cases — handling permission requests, dealing with null last-known locations, and updating the UI reactively using StateFlow. Each of these challenges forced me to read documentation carefully, test incrementally, and truly understand the code rather than just copying it.

My chosen theme is SDG 12: Responsible Consumption and Production, and it gave my project a clear social purpose that shaped every design decision. ReServe is built around the idea that surplus food and unused items in a community should not go to waste — they should be shared, borrowed, or reserved before being discarded. This directly addresses SDG 12's goal of reducing waste and encouraging sustainable consumption patterns. Features like the "Going Soon" screen (highlighting near-expiry food at discounted prices), the borrowing system for non-food items, and the community listings powered by Firebase were not just technical exercises — they were direct responses to the problem of overconsumption and unnecessary waste. The GPS feature helps users find nearby listings, making sustainable sharing practical and convenient. The Open Food Facts API integration brought real nutritional transparency to food items, encouraging users to make more informed and mindful choices about what they consume. Building ReServe taught me that mobile development is most meaningful when it serves people and the planet. SDG 12 kept me grounded throughout the project, reminding me that every feature I built had a potential real-world impact — reducing food waste, extending the life of usable items, and fostering a culture of sharing within the community rather than disposability.
