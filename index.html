1️⃣ Lab 1 — Basic Layout with Jetpack Compose

🖼️ UI Layout
▶ Watch VSR Video GitHub Repo
💡 Reflection
For this lab, I recreated the Home Screen layout of ReServe using only Row, Column, and Box, while changing the colors, fonts, and images to make it my own. I used Column to stack the main sections vertically, Row for elements that needed to sit side-by-side like the top bar icons, and Box to layer the profile image over the background banner. The trickiest part was getting the spacing and alignment right using Modifiers, but experimenting with padding(), fillMaxWidth(), and weight() helped me understand how Compose layouts work declaratively. Overall, this task gave me a solid foundation for building the rest of my e-portfolio project.

2️⃣ Lab 2 — Interaction with Jetpack Compose

🖱️ User Interaction
▶ Watch VSR Video GitHub Repo
💡 Reflection
For this lab, I upgraded my SDG app's static layout by adding a TextField and a Button to make it interactive. I used remember { mutableStateOf("") } to hold the input value, so that whenever the user types and taps the button, the state changes and Compose automatically recomposes the UI. This connected directly to the lecture concept of state hoisting — keeping the state variable at the top of the Composable so both the TextField and Button could read and update it. Once I saw recomposition happen live, the concept of reactive UI in Compose finally clicked for me.

▶ Watch VSR Video GitHub Repo

03 · Project Submissions

Projects

🚀 Project 1 — ReServe

Lab 4 + Working Features · 5+ Screens
🎯 Problem Statement

ReServe addresses food waste and underused household items within local communities by connecting people who have surplus food or unused belongings with neighbours who need them. It extends Lab 4's multi-screen navigation pattern using Jetpack Navigation Compose to link a home/browse screen, category screens, item detail screens (food and non-food), a cart/checkout flow, a listing-creation flow, chat, and a profile screen.

✨ Key Features

Five connected screens linked through Navigation Compose. A shared ReServeViewModel holds app state (items list, cart, user listings) via StateFlow. Users can add new items that immediately appear across all screens. Reservation and borrowing actions update stock/availability dynamically.

📚 Lessons Learned

Building ReServe taught the value of centralizing app state in a single ViewModel using StateFlow. Designing the navigation graph for 5+ screens reinforced how NavHost routes and arguments let separate screens share state. Lab 4's patterns scaled cleanly — the same NavHost/ViewModel/StateFlow architecture just grew in scope.

🌐 Project 2 — ReServe Advanced

Lab 5 + API + GPS + Firebase · 7+ Screens
🎯 Problem Statement

Food waste and unused household items are common in communities — edible surplus gets discarded while usable non-food items sit idle, even though nearby residents could use them. ReServe lets users list surplus food (at a discount, before it expires) and lend non-food items (tools, furniture, books) to neighbours, reducing waste and encouraging community sharing in line with SDG 12.

✨ Key Features

🗄️ Room: Six entities (UserListedItem, CartItem, ChatMessage, Reservation, BorrowedItem, UserProfile) with full MVVM + Repository architecture.

📡 REST API: Open Food Facts via Retrofit fetches live nutrition data — calories, macros, allergens, Nutri-Score — displayed in a NutritionCard on FoodDetailScreen.

📍 GPS Sensor: FusedLocationProviderClient via LocationViewModel auto-detects user location and computes real-time distance to item sellers.

☁️ Firebase Firestore: FirestoreRepository syncs community listings to the cloud whenever a user adds, updates, or deletes an item.

📚 Lessons Learned

Combining Room with Firestore required careful separation — Room as the offline-first source of truth (exposed via Flow/StateFlow), Firestore as a sync layer with graceful degradation when offline. The REST API taught the importance of defensive nullable data models. GPS integration highlighted runtime permission handling and fallback values. Overall, layering these four data sources within a shared ViewModel/Repository reinforced the value of unidirectional data flow for keeping the UI reactive and consistent.

04 · Reflection & Learning Journey

From Lab 1 to Project 2

01

My journey began in Lab 1 where I learned the very basics of Jetpack Compose — arranging UI elements using Row, Column, Box, and Text to recreate a screen layout. It felt simple but foundational. By Lab 2, I added interactivity through TextField and Button with state management. Lab 3 pushed me further into Material Design 3, where I learned to apply consistent color schemes, typography, and theming — which directly shaped the visual identity of ReServe. Lab 4 introduced Navigation and ViewModel, teaching me how to manage multi-screen flows and share state across composables without losing data on rotation. Lab 5 was a turning point — integrating Room Database taught me the full MVVM + Repository architecture. By Project 2, I had built upon every single lab: combining Room for local persistence, Firebase Firestore for cloud sync, the Open Food Facts REST API via Retrofit for live nutrition data, and GPS via FusedLocationProviderClient for real-time location awareness. What started as a static UI layout had evolved into a fully connected, sensor-aware, data-persistent community sharing application.

02

The biggest challenge I faced was understanding how all the architectural layers connect — specifically making the Room DAO, Repository, ViewModel, and StateFlow work together as a clean pipeline without causing data loss or UI inconsistencies. Debugging Room migration errors (when I added new entities like UserProfileEntity and ReservationEntity) required learning about fallbackToDestructiveMigration() and understanding database versioning. Integrating Firebase Firestore alongside Room was another challenge — I had to ensure local inserts and cloud syncs happened correctly without blocking the main thread, which taught me the importance of Kotlin coroutines and suspend functions. On the API side, parsing the Open Food Facts JSON response and gracefully handling null fields required careful state management using a sealed class. The GPS integration also had edge cases — handling permission requests, dealing with null last-known locations, and updating the UI reactively using StateFlow. Each challenge forced me to read documentation carefully, test incrementally, and truly understand the code.

03

My chosen theme is SDG 12: Responsible Consumption and Production, and it gave my project a clear social purpose that shaped every design decision. ReServe is built around the idea that surplus food and unused items in a community should not go to waste — they should be shared, borrowed, or reserved before being discarded. This directly addresses SDG 12's goal of reducing waste and encouraging sustainable consumption patterns. Features like the "Going Soon" screen (highlighting near-expiry food at discounted prices), the borrowing system for non-food items, and community listings powered by Firebase were not just technical exercises — they were direct responses to the problem of overconsumption and unnecessary waste. Building ReServe taught me that mobile development is most meaningful when it serves people and the planet, reminding me that every feature had a real-world impact: reducing food waste, extending the life of usable items, and fostering a culture of sharing over disposability.
